# ADR-003: Client-side BYOM provider abstraction, small fixed provider set for v1

**Status:** Accepted · **Date:** 2026-07-15

## Context

The vision lists seven-plus AI provider targets (OpenAI, Claude, Gemini, Groq, OpenRouter, Ollama, LM Studio, generic OpenAI-compatible endpoints). Given [ADR-001](ADR-001.md) (no mandatory backend), provider calls must be made directly from the browser using credentials the user supplies and controls.

This is viable because the major hosted providers (OpenAI, Anthropic, Groq, OpenRouter, Google Gemini) support CORS for browser-origin requests on their chat/completion endpoints, and local runtimes (Ollama, LM Studio) serve on `localhost` with configurable CORS the user enables themselves. No provider in the target list strictly requires a server-side proxy for v1.

Building and testing all seven-plus providers before anything ships is a large surface area for an MVP, and most of that surface is repetitive (request/response shape mapping) once the abstraction is right.

## Decision

Define a single `ProviderAdapter` interface (request shape in, normalized response out: text, token usage, latency, error taxonomy) that all providers implement. Ship v1 with three reference implementations, chosen to cover the three integration shapes every other provider falls into:

1. **OpenAI** — the de facto request/response shape that OpenRouter, Groq, and most "OpenAI-compatible" self-hosted endpoints also speak.
2. **Anthropic** — the shape Claude uses, meaningfully different (message format, system prompt handling, streaming events) from OpenAI's.
3. **Ollama** — the local-model, no-API-key, localhost-endpoint shape.

Gemini, Groq, OpenRouter, LM Studio, and arbitrary OpenAI-compatible endpoints are explicitly **post-v1**: adding them should require only a new adapter file, no core changes, and is the intended first-contribution task for the OSS project (see [ROADMAP.md](../../ROADMAP.md)).

Credentials are stored client-side (via the storage layer in [ADR-002](ADR-002.md)), encrypted at rest with a key derived from a user-supplied passphrase where the platform supports it, never transmitted anywhere except directly to the provider's own API.

## Consequences

- **Positive:** proves the abstraction against a hosted-cloud shape, a differently-shaped hosted-cloud shape, and a local/no-auth shape — the three axes most future providers vary on. Keeps v1 scope bounded. Gives the OSS project an obvious, well-scoped contribution path.
- **Negative:** v1 users wanting Gemini/Groq/OpenRouter/LM Studio day one don't get it; must be clear in the UI/docs that the provider list is intentionally short at launch.
- **Risk accepted:** browser-side API keys are visible to anyone with access to the user's browser storage/devtools. This is the same trust model as every other BYOK browser tool (e.g. local chat UIs). It must be disclosed explicitly in the UI, not just in docs. See [architecture.md § Security and Privacy](../architecture.md#security-and-privacy).
