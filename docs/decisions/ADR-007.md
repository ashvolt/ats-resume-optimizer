# ADR-007: Drop Google Docs / Gmail push from v1 scope

**Status:** Accepted · **Date:** 2026-07-15

## Context

F-10 (Push to Google Docs) and F-11a (Email Recruiter via Gmail) in the v0.1 [FEATURES.md](../../FEATURES.md) work by sending a prompt to Claude via `window.sendPrompt`, which then uses Claude.ai's own connected Google Drive/Gmail MCP tools to act on the user's behalf. [ADR-001](ADR-001.md) removes the Claude Artifact runtime this depends on.

Replacing it natively would mean implementing real Google OAuth (Drive + Gmail scopes) from the standalone app. This is technically possible client-side (Google Identity Services supports a PKCE/implicit flow with no backend token exchange required for this use case), but it means: registering and maintaining a Google Cloud OAuth client, handling Google's app verification process for sensitive scopes (Gmail send, Drive write) which is a nontrivial review process for an open-source project with no company entity behind it, and building/testing an integration surface unrelated to the core score → optimize → export loop.

## Decision

**Drop Google Docs / Gmail push from v1.** v1 export is Markdown and PDF only ([features/export-engine.md](../features/export-engine.md)), generated and downloaded entirely client-side. Google Docs/Gmail integration is moved to the future roadmap ([ROADMAP.md](../../ROADMAP.md)), to be redesigned (client-side OAuth per this ADR's Context, or otherwise) only once the core rearchitecture is stable and there's appetite to deal with Google's OAuth verification process.

The existing `window.sendPrompt` code path in `ats-resume-builder.jsx` is deprecated, not deleted immediately — see [CHANGELOG.md](../../CHANGELOG.md).

## Consequences

- **Positive:** removes a large, OAuth-verification-gated feature from the critical path to shipping v1; keeps v1 export simple and fully offline-capable.
- **Negative:** v0.1 users relying on the Google Docs push workflow lose it until the future-roadmap version ships; the recruiter-email-detection logic (F-11a) also goes with it and would need to be re-scoped if/when this returns.
- **Follow-up:** when revisited, evaluate whether Gmail "send" scope is worth the OAuth verification burden versus a lower-friction "open a pre-filled mailto: link" fallback that needs no OAuth at all.
