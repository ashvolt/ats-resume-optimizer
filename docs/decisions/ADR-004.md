# ADR-004: Hybrid ATS engine — deterministic core, optional AI enhancement layer

**Status:** Accepted · **Date:** 2026-07-15

## Context

The v0.1 prototype's ATS score is 100% deterministic keyword-frequency matching, computed client-side with no AI call. It has known, documented bugs ([DEVELOPMENT_PLAN.md](../../DEVELOPMENT_PLAN.md) Phase 1: bigrams cross sentence boundaries, tech terms fail to match due to trailing punctuation, no JD section weighting) but the underlying approach — auditable, instant, free, deterministic — is sound and was never the actual problem.

The vision asks whether the ATS engine should be "redesigned" and wants an evaluation of the current approach, plus richer output: explanations for every deduction, formatting/readability/action-verb/impact-metric evaluation, not just keyword coverage.

Two options were considered: (a) replace scoring with an LLM call that reads the JD and resume and returns a score + explanation, or (b) keep and finish the deterministic engine as the core, and add AI as an optional second pass.

Option (a) fails [vision.md](../vision.md)'s "cost-efficient" and "works with zero AI configured" principles outright — it would make the core score → gap → fix loop require a paid API call every time, and non-deterministic scoring makes "did my edit help?" unreliable (same input, two different scores).

## Decision

**Keep and finish the deterministic engine as the scoring core.** This means executing the already-designed but unimplemented plan: sentence-aware tokenization, tech-term normalization/whitelist, JD section weighting (Requirements 2×, Benefits 0×), and keyword categorization — see [DEVELOPMENT_PLAN.md](../../DEVELOPMENT_PLAN.md) §1 and [features/ats-engine.md](../features/ats-engine.md) for the full spec. This layer also owns everything rule-detectable without semantic understanding: section completeness, weak-verb detection, missing quantified-impact bullets, formatting hazards (tables, images, non-standard headers) that break ATS parsers.

**Add AI as an optional, clearly-separated second pass**, active only when a provider is configured ([ADR-003](ADR-003.md)): it evaluates things the rule engine structurally cannot — whether a bullet's phrasing is actually strong, whether "led a team" is substantiated, relevance of experience to the specific role's seniority/domain, and generates the rewrite suggestions in [features/resume-editor.md](../features/resume-editor.md). AI-layer output is always additional detail attached to a deterministic-engine finding, or a separate "AI insights" section — it never silently overrides or replaces a deterministic score.

## Consequences

- **Positive:** the tool's core value (score, gaps, why) works with zero API cost and zero setup, exactly like v0.1's pitch. Every deduction traces to a named rule, which is what makes the "why" explanation in Step 4 of the vision possible without hallucination risk. Reproducible scores make the "before → after" progression in Step 7 trustworthy.
- **Negative:** the deterministic engine cannot evaluate true semantic relevance (e.g., "this candidate's fintech experience is relevant to this fintech JD" beyond keyword overlap) — that gap is real and is explicitly the AI layer's job when a provider is present, and explicitly a known limitation when it isn't.
- **Follow-up:** the existing DEVELOPMENT_PLAN.md Phase 1 tokenizer/section-weighting fixes become v1 requirements, not nice-to-haves — they're prerequisites for the score being trustworthy at all.
