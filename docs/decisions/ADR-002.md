# ADR-002: Local-first persistence — IndexedDB, exportable, no mandatory server

**Status:** Accepted · **Date:** 2026-07-15

## Context

The vision asks for version history, an audit trail of AI suggestions, session persistence across page reloads, and multi-JD comparison — all of which imply *some* durable storage. [ADR-001](ADR-001.md) rules out a mandatory backend. The open question was whether v1 needs a real server + database to deliver these, or whether browser-local storage is sufficient.

Given the "privacy-first" principle and that the primary user (v1) is an individual optimizing their own resume on their own machine, requiring an account and a server round-trip to persist a resume draft is disproportionate — and directly conflicts with "user data remains under user control."

## Decision

All application state (Resume JSON, JD records, suggestion history, version history, provider settings, encrypted API keys) is persisted in the browser via **IndexedDB**, wrapped behind a storage interface (not called directly from feature code, so a future sync backend can implement the same interface — see Future work below).

Every persisted entity must be exportable as a file (JSON for structured data, Markdown for resume content) and importable back in, so a user's data is never trapped in one browser profile.

No account system, no login, and no server-side database in v1.

## Consequences

- **Positive:** zero infra cost, zero data-breach surface for us to create, works offline after first load, trivially matches "no signup."
- **Negative:** no cross-device sync in v1; clearing browser storage loses history unless the user exported it; no server-side backup. These are acceptable for the stated v1 audience (individual, single-device user) and should be surfaced in the UI (e.g., prompt users to export before clearing data).
- **Follow-up (explicitly deferred, not designed in v1):** the storage interface should be written so that a self-hostable sync backend (for contributors/teams wanting multi-device) can be added later by implementing the same interface server-side, without changing feature code. This is a seam to leave open, not a v1 deliverable.
